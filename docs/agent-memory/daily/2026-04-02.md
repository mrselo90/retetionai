## Context
- Main focus: super admin tooling for WhatsApp and order-flow testing.
- Environment: local workspace, Shopify shell + admin panel + API integration review.

## Decisions
- Create a dedicated super admin test screen instead of relying on ad hoc API calls.
- Put the feature in the super admin panel, not Shopify shell. Reason: the real WhatsApp and order-processing logic lives in `packages/api`, while the Shopify app is only an embedded operations shell.
- Reuse existing production-safe helpers instead of hand-writing DB records. Test orders should flow through `processNormalizedEvent`, and AI replies should flow through the same conversation + `generateAIResponse` path.
- Keep two testing layers in the screen:
  - realistic order creation + delivered trigger
  - simulated inbound WhatsApp reply with optional live send of the AI response
- Keep `welcome template` as a merchant-facing feature, but make it provider-template aware instead of plain free text only.
- New welcome-message policy:
  - if a 24h WhatsApp conversation window is open, send rendered free text
  - if the window is closed and sender is Twilio, send approved template via `ContentSid + variables`
  - if Twilio window is closed and no template is configured, fail clearly instead of silently sending unsupported free text
- Refined the welcome-template product decision:
  - merchants should not manage `Twilio Content SID`
  - merchants should only compose the human-readable welcome body with selectable customer/order variables
  - Twilio approved template identity must be platform-managed

## Observed Issues
- Shopify shell does not expose a first-class WhatsApp testing screen.
- Existing API test helpers appear to exist outside the mounted production route surface.
- `packages/api/src/routes/test.ts` contains useful WhatsApp and event simulation logic, but it is not mounted in `packages/api/src/index.ts`, so super admin needed its own safe production route surface.
- Shell-side conversation reply is not enough for setup testing because it requires an already-existing user/conversation/order context.
- Real inbound WhatsApp still depends on provider webhook signatures and worker processing. The new admin simulate route bypasses that only for controlled AI-flow testing.
- Effective WhatsApp sender can differ from merchant expectation because sender resolution comes from `getEffectiveWhatsAppCredentials(...)`, which may use merchant integration creds or corporate creds depending on merchant setup.
- Real production incident found: inbound WhatsApp replies were received but skipped with `Could not resolve merchant for incoming WhatsApp message`.
- Root cause: Twilio webhook parsing did not carry the recipient business number into merchant lookup, so `phoneNumberId` stayed `null` and inbound replies could not be attached to a merchant.
- Fix applied: for Twilio inbound webhooks, reuse the recipient `To` number as the merchant lookup key and normalize `whatsapp:` prefixes during matching.
- New production issue (resolved):
  - inbound webhook ingestion succeeded but async processing failed in worker due to wrong internal API base URL selection
  - this caused no AI response after customer reply and dead-letter queue entries for WhatsApp inbound jobs
